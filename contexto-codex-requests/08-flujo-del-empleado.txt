FLUJO DEL EMPLEADO

OBJETIVO
Permitir que un empleado cree y consulte requisiciones sin login tradicional, pero manteniendo identificación persistente.

1. Primer acceso
- El empleado entra a la aplicación.
- El frontend busca employeeToken en localStorage.
- Si no existe, redirige a /employee/identify.

2. Identificación
- El empleado llena:
  - Nombre.
  - Departamento.
  - Código de empleado opcional.
  - Teléfono o extensión.
- El frontend envía POST /api/employees/identify.
- El backend crea o recupera empleado.
- El backend responde con publicToken.
- El frontend guarda publicToken en localStorage.

3. Acceso posterior
- Al volver a abrir la aplicación, se lee publicToken.
- Se llama GET /api/employees/me.
- Si el token es válido, se muestra el home.
- Si no es válido, se limpia localStorage y se redirige a identificación.

4. Crear requisición
- El empleado entra a Crear requisición.
- Selecciona prioridad.
- Agrega comentario general.
- Agrega uno o varios materiales.
- Puede seleccionar material del catálogo o escribir material manual.
- Define cantidad y unidad.
- Confirma antes de enviar.
- El frontend llama POST /api/requisitions con x-employee-token.

5. Al enviar requisición
- Backend valida empleado.
- Backend crea código automático.
- Backend crea requisición e items dentro de una transacción SQL.
- Backend registra historial CREATED.
- Backend crea notificación para Compras.
- Backend emite Socket.IO requisition:created.
- Frontend muestra confirmación.

6. Seguimiento
- El empleado puede entrar a Mis requisiciones.
- Ve listado de sus solicitudes.
- Puede filtrar por estado y fecha.
- Puede buscar por código.

7. Detalle
- El empleado abre una requisición.
- Ve:
  - Código.
  - Estado.
  - Fecha.
  - Prioridad.
  - Materiales.
  - Cantidades solicitadas/aprobadas/entregadas.
  - Comentarios.
  - Historial de estados.

8. Tiempo real
- El empleado se conecta a Socket.IO con employeeToken.
- El backend lo une a room employee:{id}.
- Cuando Compras cambia estado, el empleado recibe requisition:statusChanged.
- Cuando Compras comenta, recibe comment:created.

9. Comentar
- El empleado puede escribir comentarios en una requisición propia.
- El comentario se guarda y se notifica a Compras.

10. Cancelar requisición
- El empleado puede cancelar si no está en estado final.
- Debe escribir motivo.
- El sistema cambia estado a Cancelada.
- Se registra historial.
- Se notifica a Compras.

11. Cambiar identidad
- Desde perfil, el empleado puede limpiar su identidad local.
- Esto elimina employeeToken del navegador.
- No elimina el registro en la base de datos.
