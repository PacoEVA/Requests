REGLAS DE NEGOCIO

1. Identificación del empleado
- El empleado no tiene login.
- Se identifica mediante un PublicToken guardado en el navegador.
- El PublicToken se guarda en localStorage en el frontend.
- El backend valida el PublicToken para saber qué empleado realiza la acción.
- Si no hay PublicToken, el empleado debe pasar por la pantalla de identificación.

2. Creación de requisiciones
- Una requisición debe tener al menos un item.
- Cada item debe tener cantidad mayor que 0.
- Un item puede venir del catálogo o ser manual.
- Si viene del catálogo, debe tener MaterialId.
- Si es manual, debe tener ManualMaterialName.
- El estado inicial siempre es Pendiente.
- Al crear la requisición se genera código automático.

3. Código de requisición
- Formato recomendado: REQ-YYYY-000001.
- El contador debe ser incremental por año.
- El código debe ser único.

4. Estados
Estados permitidos:
- Pendiente.
- En revisión.
- Aprobada.
- Rechazada.
- En compra.
- Lista para entregar.
- Entrega parcial.
- Entregada.
- Cancelada.

Estados finales:
- Rechazada.
- Entregada.
- Cancelada.

5. Transiciones recomendadas
- Pendiente -> En revisión.
- Pendiente -> Aprobada.
- Pendiente -> Rechazada.
- Pendiente -> Cancelada.
- En revisión -> Aprobada.
- En revisión -> Rechazada.
- En revisión -> Cancelada.
- Aprobada -> En compra.
- Aprobada -> Lista para entregar.
- Aprobada -> Entregada.
- En compra -> Lista para entregar.
- Lista para entregar -> Entregada.
- Lista para entregar -> Entrega parcial.
- Entrega parcial -> Entregada.
- Entrega parcial -> En compra.

6. Cancelación por empleado
- El empleado puede cancelar mientras la requisición no esté en estado final.
- Si la requisición ya fue entregada, rechazada o cancelada, no se puede cancelar.
- Debe registrarse motivo.

7. Rechazo
- Solo usuario interno puede rechazar.
- Requiere motivo obligatorio.
- Debe notificar al empleado.

8. Aprobación
- Usuario interno con rol Admin o Compras puede aprobar.
- Supervisor puede aprobar solo si se activa flujo por supervisor.
- Se puede aprobar cantidad menor a la solicitada.

9. Entrega
- Al marcar como entregada, registrar cantidades entregadas.
- Descontar stock local al entregar, según la regla definida para inventario.

10. Inventario
- El inventario se administra localmente en la base de datos Requests.
- Se permite ajuste manual de stock local desde el dashboard privado.
- Stock mínimo se mantiene localmente.

11. Comentarios
- Todos los comentarios deben guardarse.
- No se pueden eliminar inicialmente.
- No incluir archivos adjuntos.
- Al crear comentario, notificar a la contraparte.

12. Dashboard
- Las métricas deben calcularse desde la base de datos.
- Al crear o actualizar requisición, refrescar contadores en tiempo real.

13. Estadísticas
- Top departamentos por cantidad de requisiciones.
- Top materiales por cantidad total solicitada.
- Requisiciones por estado.
- Requisiciones por prioridad.
- Tiempos promedio.

14. Seguridad
- Los endpoints admin requieren JWT.
- Los endpoints de empleado requieren x-employee-token.
- No exponer datos de otros empleados en área pública.
- Validar todo en backend.

15. Recuperación de contraseña
- El Admin puede restablecer contraseña desde Usuarios.
- El usuario debe cambiarla al iniciar sesión si RequirePasswordChange = true.
- No requiere email en primera versión.

16. Eliminaciones lógicas
- Usuarios, departamentos y materiales se desactivan con IsActive = 0.
- No borrar registros con historial asociado.

17. Restricciones eliminadas
- No implementar sucursales.
- No implementar adjuntos.
- No implementar configuración de nombre de empresa.
- No implementar configuración dinámica de estados.
- No implementar rol Consulta.
